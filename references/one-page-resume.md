# 一页简历压缩规则

一页版不是默认产物。仅在以下情况生成：

1. 用户明确说需要一页版
2. 整版重写后用户确认"需要"
3. 用户原始请求本身就是压缩到一页

整版重写后主动问一句：

```text
要不要我顺手再给你压缩成一页版简历？
```

## 压缩原则

1. **只保留最能支撑目标岗位的经历和成果**——一页版目标是"提高面试转化率"，不是"完整存档"。
2. **优先保留近 5-8 年、与 JD 最相关的内容**。
3. **删除**：低价值自评、弱相关项目、重复技能、低信息量 bullet。
4. **每条经历压到 2-4 条高密度 bullet**。
5. **每个项目仍必须保留一句话项目描述**，可压到一行，但不能省略——没有上下文的 bullet 在一页简历中更致命。
6. **摘要压到 1-2 行**，只留价值主张 + 核心量化。
7. **技能压到一行**，只列与目标岗位最相关的 8-10 项。

## 压缩后必须保留

- 联系信息（精简到一行）
- 价值主张（1-2 行）
- 最近 2-3 段经历（每段项目描述 + 2-4 bullet）
- 核心技能（一行）
- 教育背景（一行，社招可极简）

## 压缩后可删

- 5 年前的经历（除非与目标岗位强相关）
- 与目标岗位无关的项目
- 课程/社团（社招删除）
- 自我评价
- 兴趣爱好

## 输出

文件保存到 `resume/resume-one-page.md`，再按 `export-templates.md` 转目标格式。

## 示例

原文 2 页 → 压缩后 1 页：

```text
张三 | 高级后端工程师 | 5 年经验
邮箱 | 电话 | GitHub
---
价值主张：主导过高并发支付系统，接口延迟优化 75%，支撑日均 50 万笔交易。

工作经历
A 公司 高级后端工程师 (2022-至今)
- 支付核心系统：面向 C 端用户的支付网关，日均 50 万笔交易
  - 主导模块重构，接口延迟 800ms→200ms，故障率降 60%
  - 设计对账机制，资金差异从月均 5 笔降到 0
B 公司 后端工程师 (2020-2022)
- 订单系统：服务 200 万用户的电商订单链路
  - 落地分库分表，支撑双十一 10 倍流量

技能：Go | MySQL | Redis | Kafka | K8s | 分布式系统
教育：XX大学 计算机科学 本科 2016-2020
```
