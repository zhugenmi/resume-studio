# 简历长度版本取舍规则

> 本文件管「长度维度」（详尽/标准/一行）。另有「雇主版本维度」（soe/internet/foreign）管评分标准与语言，见 `employer-variants.md`。两维正交：先定雇主版本，再在该版内按篇幅取长度。工作版文件名为 `resume-<variant>.md`。

`resume-<variant>.md` 工作版**必含三个长度版本**，用户按简历剩余篇幅选用。同一份简历只投一个版本，不要混用。

## 三个版本

### 详尽版（主力项目，5-6 条 bullet）

- 适用：作为简历里唯一/最重项目，或专门为某岗位定制一页项目深挖版。
- 结构：项目元信息行（角色·时间·技术栈）+ 一句话项目描述 + 5-6 条 STAR bullet + 结果行。
- 每条 bullet：`强动词 + 动作 + 产物 + 结果（含量化）`。
- 结果行：端到端量化（覆盖规模 / 产出 / 耗时 / 成本 / 提升幅度）。

### 标准版（2-3 个项目并列，3 条 bullet）

- 适用：简历有 2-3 个项目并列展示，每段需要克制篇幅。
- 结构：项目元信息行 + 一句话项目描述 + 3 条高密度 bullet + 结果行（一行）。
- 取舍：保留最强量化、最相关 JD、最能体现架构深度的 3 条，其余删除。

### 一行版（项目栏紧凑展示）

- 适用：简历篇幅紧张，项目只作存在感展示；或次要项目。
- 结构：`项目名 · 一句话定位 · 核心技术 · 规模数字`，单行。
- 不含 bullet，靠定位句 + 规模数字传递价值。

## 版本取舍表

| 简历剩余篇幅 | 目标项目地位 | 选用 |
|-------------|------------|------|
| ≥半页 | 唯一主力项目 | 详尽版 |
| 1/3 页 | 2-3 项目之一 | 标准版 |
| <1/3 页 | 次要项目 | 一行版 |

## 共同要求（三版本都守）

- **项目描述上下文先行**：详尽版/标准版必须有一句话项目描述（系统定位 + 服务对象 + 业务问题），一行版用定位句承担。
- **不编造数字**：缺数据标 `[量化指标待补：...]`，不脑补。
- **强动词开头**：负责→主导/牵头，参与→协同，做了→重构/落地（见 `star-rewrite.md`）。
- **JD 关键词融入**：有 JD 时，硬技能词原样出现在 bullet 里，ATS 命中。

## 投岗侧重表（可选，附在 resume.md 末尾）

附一张「投不同岗位时应强化的 bullet」表，帮用户按 JD 快速切换重点：

```markdown

|--------|----------------|

| 后端工程师 | {{...}} |
| 全栈工程师 | {{...}} |

```

## 待补清单（必附）

`resume.md` 末尾附「待补清单」——列出所有 `[待补：...]` 占位符，写死前必须填实。包括但不限于：角色、时间、GitHub 链接、是否上线/有真实用户、运行时数据（覆盖量/耗时/成本）。

> 区分：代码可证的事实（节点数、平台数、代码行数、接口数）可写死；运行时数据（论文数、字数、耗时、token 成本）必须真实跑一次再填。
